Architecture Approach

How should the schema analysis feature be structured?

A

Client-Side Analysis (Recommended)

All similarity detection, recommendations, and migration generation run in the browser. API routes only serve mock data (or fetch schema from projects in production). The analysis engine lives in lib/schema-analysis/ as pure functions.

Pros

  • No server load for analysis computation
  • Instant feedback as user adjusts filters
  • Simpler API surface (just data serving)
  • Easier to test (pure functions)
  • Matches React Query pattern (fetch data, transform client-side)

Cons

  • Large schemas may cause UI jank (mitigated by web workers)
  • Analysis results not shareable via URL without re-running
B

Server-Side Analysis

API routes do the heavy lifting — similarity detection, recommendations, migration generation all happen server-side. Client just renders results.

Pros

  • Consistent results regardless of client device
  • Can cache analysis results server-side
  • Analysis results shareable via API

Cons

  • More API routes, more server complexity
  • Slower iteration (every parameter change = API call)
  • Harder to test (need API mocking)
  • Doesn't match existing codebase patterns as well
C

Hybrid (Spec's Approach)

Follow the spec exactly: API routes for each operation (/analyze, /matrix, /recommendations, /generate-migration, /apply-migration). Analysis logic lives in lib/ but is invoked by API handlers.

Pros

  • Matches spec 1:1
  • API routes are thin wrappers around lib functions
  • Can switch between mock/real data at API layer
  • Migration apply needs server-side anyway

Cons

  • More boilerplate than pure client-side
  • Some routes feel unnecessary for client-only analysis
